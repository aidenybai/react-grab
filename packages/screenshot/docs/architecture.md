# Architecture Documentation

This document covers how @react-grab/screenshot's internals work. It's intended to aid in understanding the code, in understanding the tricks the library uses to reproduce a live DOM subtree pixel-for-pixel inside an SVG `<foreignObject>`, and hopefully to enable people to modify the code. Many of the techniques described here are inherently fragile because they sit on top of browser behaviors that were never specified for this purpose: computed-style enumeration order, `foreignObject` rasterization quirks, canvas tainting rules, and per-engine differences in how UA default styles are applied. The current approach is not necessarily the best or only way to do these things, and some of these details will need to change as browsers evolve.

### Design principles

- **Use the browser's renderer instead of reimplementing paint**
  The capture is produced by serializing a styled clone of the target into `<svg><foreignObject>` and letting the browser's own layout and paint engine rasterize it. Libraries that re-draw the DOM onto a canvas by hand (html2canvas) have to reimplement text shaping, gradients, filters, stacking contexts, and every future CSS feature; the `foreignObject` approach gets all of that for free and only has to make sure the clone _looks_ identical to the live tree. Almost everything in this codebase is in service of that one goal: freezing computed styles, live form state, scroll positions, and media frames into a self-contained document that renders the same in isolation.
- **Diff against UA defaults so the output stays small**
  `getComputedStyle` returns ~350 longhand properties per element, and serializing all of them makes captures huge and slow to parse. Instead, each element's snapshot is diffed against a baseline computed for a bare element of the same tag in a sandbox iframe ([capture/default-styles.ts](../src/capture/default-styles.ts)), so only non-default declarations survive. Elements with identical diffed styles then share a single generated class ([capture/style-registry.ts](../src/capture/style-registry.ts)). The cost of this principle is a family of correctness exceptions - properties that must be emitted even when they equal the baseline - documented in the notes below.
- **Read-only pass over the live DOM**
  The capture never mutates the target subtree. The read pass ([capture/snapshot-styles.ts](../src/capture/snapshot-styles.ts)) only calls `getComputedStyle` and reads scroll offsets; all writes happen on detached clones or in the off-screen sandbox iframe. This keeps `captureNode` safe to call against a live application without visual flicker, layout thrash on the page, or interference with framework-managed DOM.
- **Decode once, rasterize many**
  One serialized SVG becomes one decoded `HTMLImageElement`, shared by every output format ([capture/rasterize.ts](../src/capture/rasterize.ts)). `toCanvas`, `toBlob`, and `toPngDataUrl` all reuse the same decode promise, so calling several output methods doesn't repeat the expensive decode-and-settle step.
- **Test against ground truth, not golden files**
  The fidelity harness ([e2e/fidelity.spec.ts](../e2e/fidelity.spec.ts)) compares every capture against a native Chromium screenshot of the same element taken in the same page state, rather than against checked-in golden images. Golden files drift with browser updates and encode the library's own bugs; a native screenshot is, by definition, what the capture is supposed to look like.

## Overview

The whole pipeline is orchestrated by `captureNode` in [index.ts](../src/index.ts). It runs as a sequence of passes:

1. **Wait and measure.** `document.fonts.ready` is awaited so font metrics are final, then the capture rect is computed. `offsetWidth`/`offsetHeight` (untransformed layout pixels) are preferred over `getBoundingClientRect()`, because the bounding rect is post-transform while every descendant is frozen at its layout size; the bounding box is only a fallback for non-HTML roots (e.g. inline SVG), rounded out to integers.
2. **Read pass.** [capture/snapshot-styles.ts](../src/capture/snapshot-styles.ts) walks the _composed_ tree (shadow roots and slot assignment resolved) and snapshots `getComputedStyle` for every element as a plain object, plus `::before`/`::after`/`::first-letter` pseudo styles (and `::marker` for list items) when a stylesheet preflight says the document defines any ([capture/pseudo-elements.ts](../src/capture/pseudo-elements.ts)), plus each element's scroll offsets. Non-painting properties (`cursor`, `will-change`, animation/transition longhands, custom properties) are skipped via a memoized predicate ([utils/should-skip-style-prop.ts](../src/utils/should-skip-style-prop.ts)). Rather than reading all ~400 longhands per element, the read pass enumerates the property names that author CSS, `@keyframes`, WAAPI animations, inline styles, and shadow-root stylesheets could possibly set ([capture/relevant-style-props.ts](../src/capture/relevant-style-props.ts)) plus an always-read core (`ALWAYS_SNAPSHOT_STYLE_PROPS`: everything the pipeline inspects, UA state-driven styling, and engine-resolved layout values), and only reads those — typically cutting `getPropertyValue` calls by ~60-70%. Properties outside that set can only hold the sandbox-baseline value, so skipping them cannot change the diff. SVG elements (presentation attributes) and form controls (attribute/state-driven UA styling) always get full snapshots, and any cross-origin-inaccessible stylesheet disables the subset for the whole capture since its rules could set anything.
3. **Nested iframe capture.** Same-origin iframes are captured recursively - `captureNode` is called on the iframe's own `documentElement` and the result is stored as a PNG data URL, later swapped in for the iframe box ([index.ts](../src/index.ts), `captureIframeContents`). A module-level `activeCaptureDocuments` set breaks recursion cycles. Cross-origin iframes first try the `resolveIframeContent` option (a user hook returning an image data URL, measured via `Image.decode`), then a `postMessage` bridge ([capture/iframe-bridge.ts](../src/capture/iframe-bridge.ts)): the framed page opts in by calling `enableIframeBridge()`, the parent posts a shape-validated capture request ([utils/parse-iframe-bridge-messages.ts](../src/utils/parse-iframe-bridge-messages.ts)) keyed by a random request id, and the frame answers with a PNG data URL plus its document background color. Requests time out to the flat gray placeholder, an in-flight flag on the bridge side breaks mutual-embedding recursion, and responses are matched by `event.source` and request id so unrelated messages are ignored.
4. **Backdrop-filter baking.** SVG `foreignObject` cannot evaluate `backdrop-filter` (the filtered backdrop lives outside the fragment being rasterized), so elements carrying one get it baked in ([capture/bake-backdrop-filters.ts](../src/capture/bake-backdrop-filters.ts)): the capture recurses once with those elements hidden (`visibility: hidden`, guarded by `skipBackdropFilterBaking` so the underlay pass never recurses again), the resulting canvas is cropped to each element's box expanded by the filter's visual extent ([utils/compute-filter-extent.ts](../src/utils/compute-filter-extent.ts)) so edge pixels blur with real neighbors, the crop is run through `CanvasRenderingContext2D.filter` with the element's `backdrop-filter` value, and the filtered PNG is layered under the element's own background layers ([utils/apply-baked-backdrop-background.ts](../src/utils/apply-baked-backdrop-background.ts)) - the authored `background-color` becomes a `linear-gradient` layer above the baked crop to preserve native paint order - while `backdrop-filter` itself is dropped from the emitted styles.
5. **Diff pass.** Each snapshot is diffed against the sandbox baseline ([capture/diff-styles.ts](../src/capture/diff-styles.ts)), the size freezing/softening policy is applied, the root element gets neutralizing overrides (margins zeroed, `position` demoted to `relative`, `transform: none`, frozen width/height), and escaped bottom margins are transferred to parents ([capture/margin-collapse.ts](../src/capture/margin-collapse.ts)). The diffed declarations register into the style registry, which returns a shared `rgs-N` class name per unique style signature.
6. **Clone pass.** [capture/clone-tree.ts](../src/capture/clone-tree.ts) deep-clones the composed tree with live state baked in: form values reflected as attributes, `<img>` frozen to its `currentSrc`, `<canvas>`/`<video>` frames rasterized, scrolled containers wrapped in a translated wrapper ([capture/scroll-offsets.ts](../src/capture/scroll-offsets.ts)), sticky descendants pinned ([capture/freeze-sticky.ts](../src/capture/freeze-sticky.ts)), and everything sanitized for XML serialization.
7. **Reference and resource inlining.** [capture/inline-svg-defs.ts](../src/capture/inline-svg-defs.ts) imports `<use>` targets and `url(#id)` references that live outside the captured subtree (including external sprite files). [capture/inline-resources.ts](../src/capture/inline-resources.ts) rewrites every external URL - `<img src>`, SVG `<image href>`, and `url(...)` inside registered style rules - to a data URL. [capture/embed-fonts.ts](../src/capture/embed-fonts.ts) emits `@font-face` CSS with inlined sources for the font families the capture actually uses.
8. **Serialize.** [capture/serialize-svg.ts](../src/capture/serialize-svg.ts) wraps the clone and the generated stylesheet in `<svg><style>…</style><foreignObject><div xmlns="…xhtml">…</div></foreignObject></svg>` via `XMLSerializer`. The wrapper `<div>` gets `all:initial` plus the frozen capture size, and a small normalization stylesheet hides scrollbars and pins `-webkit-text-size-adjust` (iOS WebKit re-applies text autosizing while rasterizing `foreignObject` content; an inline style is expanded too late, so it must be a rule).
9. **Rasterize.** [capture/rasterize.ts](../src/capture/rasterize.ts) returns a `CaptureResult` whose output methods lazily decode the SVG once and draw it to a canvas at `scale × pixelRatio`.

The sandbox iframe is created lazily during the diff pass and disposed in a `finally`, so a throwing capture never leaks the iframe into the host page.

## Notes about the default-style baseline

The diffing strategy only works if the baseline actually matches what the browser would apply to an unstyled element in the capture. [capture/default-styles.ts](../src/capture/default-styles.ts) computes baselines by mounting a bare probe element inside a hidden same-origin `<iframe>` appended to the host document.

The sandbox iframe is hidden by positioning it off-screen (`position:absolute; left:-9999px; visibility:hidden`), never with `display:none` - Firefox computes garbage styles inside display-none iframes ([Mozilla bug 548397](https://bugzilla.mozilla.org/show_bug.cgi?id=548397)). Baselines are cached in a module-level FIFO cache keyed by namespace, tag name, `<input>` type, pseudo selector, and font size. Font size is part of the key (and is set on the probe) because many UA defaults are `em`-relative - margins on headings, padding on form controls - so a probe at the default 16px would produce wrong baselines for elements with a different inherited font size. SVG probes are mounted inside a scratch `<svg>` container since bare SVG leaf elements compute differently outside one.

Equality with the baseline is not always safe grounds for omission, which is what the exception lists in [constants.ts](../src/constants.ts) encode:

- `CONCRETE_VALUE_STYLE_PROPS` are always emitted. `width`/`height` compute to concrete pixel values on the probe that have nothing to do with the element's layout. `box-sizing` is subtler: Chromium's border-box theming for form controls comes from LayoutTheme style adjustment, which does not run for `foreignObject` content rasterized through an `<img>`, so an omitted equal-to-baseline `box-sizing` regresses to `content-box` there. Border widths are included because a `0px` width produced by `border-style: none` matches the baseline and would be dropped - letting an emitted `border-style` resurrect the UA default `medium` (3px) width.
- `CURRENTCOLOR_DEFAULT_STYLE_PROPS` (border colors, `text-decoration-color`, `outline-color`, …) have `currentColor` as their initial value, so baseline equality is meaningless: the sandbox resolves `currentColor` against a different `color`. They are instead emitted iff they diverge from the element's own `color` - and omitted when equal, which also dodges a Chromium quirk where an explicit `-webkit-text-fill-color` on the element suppresses `::first-letter` color.
- Inherited properties get a second chance in `diffStyles`: if a parent _emitted_ an inherited property (because it diverged from the parent's baseline) but the child's value happens to equal the child's baseline, the child would silently inherit the parent's emitted value inside the capture. So any inherited property present in the parent's emitted set is re-emitted on the child whenever the child's value differs from the parent's.

## Notes about width freezing and softening

Freezing every element's used `width`/`height` is what makes the clone reproduce the live layout without re-running the page's stylesheets, but hard-freezing everything breaks the boxes whose size is supposed to be derived. `applySizeFreezingPolicy` in [capture/diff-styles.ts](../src/capture/diff-styles.ts) carves out the exceptions:

- **Inline non-replaced elements** get their size properties deleted entirely - inline boxes size to content, and a frozen width on a `<span>` is ignored at best and harmful when the diff also changes text metrics.
- **Table boxes** (`table`, `tr`, `td`, …) get frozen sizes _softened_ rather than dropped or kept. Table box sizes come out of the table layout algorithm; freezing them as hard values grows captioned tables by the caption height and re-wraps cells (the snapdom #209/#429/#434 class of bugs). So all frozen size properties are removed, and for `table`/`td`/`th` the captured width is re-emitted as a `min-width` floor instead - unless the author already set a real `min-width` - letting the table algorithm reproduce the observed geometry without over-constraining it.
- **Flex items** get `flex-grow: 0; flex-shrink: 0; flex-basis: auto` so the frozen width _is_ the flex-resolved size rather than an input to a second flex resolution, and flex/grid items with `min-width: auto` get `min-width: 0` so frozen sizes aren't fought by the implied minimum.

The root element additionally gets `applyRootStyleOverrides`: margins and inset offsets are zeroed, `fixed`/`absolute`/`sticky` positioning is demoted to `relative`, `transform`/`float`/`clear` are neutralized, and `box-sizing: border-box` plus the frozen capture rect are pinned. The capture renders the target at its untransformed layout size - the root's own transform and any ancestor transforms are deliberately not applied.

A related fix-up lives in [capture/margin-collapse.ts](../src/capture/margin-collapse.ts): freezing computed heights gives every box a definite height, which stops last-child bottom margins from collapsing through their parent (CSS 2.1 §8.3.1 requires an `auto` height for margin escape). `applyEscapedBottomMarginTransfers` detects parents whose live border-box bottom coincides (within 0.6px) with their last in-flow block child's bottom - the geometric signature of an escaped margin - and re-emits the escaped margin on the parent so the following sibling stays where it was.

## Notes about style signature dedup

[capture/style-registry.ts](../src/capture/style-registry.ts) collapses identical style sets into shared classes. The signature is the sorted `prop:value;` join of the base declarations concatenated with the `::before`, `::after`, and `::first-letter` declaration blocks; the first registration of a signature allocates the next `rgs-N` class name and every later match reuses it. `toCssText` emits one rule per unique class (plus its pseudo rules) into the single `<style>` element inside the SVG. On real pages - long lists, tables, repeated cards - this is the difference between hundreds of duplicated declaration blocks and a handful of classes.

Original `class` attributes are stripped from clones entirely (all styling comes from the snapshots), as are `style` attributes. For SVG elements there is one more step: paint presentation attributes (`fill`, `stroke`, `opacity`, …) are removed from clones that received a generated class. Presentation attributes sit _below_ class rules in the cascade, but the diffed class omits properties equal to the UA default - so a stale attribute (e.g. `fill="red"` overridden by CSS on the live page) would win inside the capture if left in place.

## Notes about cloning the composed tree

[capture/clone-tree.ts](../src/capture/clone-tree.ts) rebuilds the target as a flat light-DOM tree that a `foreignObject` can render without shadow roots, scripts, or live state.

**Shadow DOM and slots.** Both the read pass and the clone pass iterate children through [utils/get-composed-child-nodes.ts](../src/utils/get-composed-child-nodes.ts): if an element has a shadow root, its shadow children are walked instead of its light children, and `<slot>` elements are replaced by their flattened assigned nodes (falling back to the slot's own children when nothing is assigned). The result is the composed tree - what the browser actually renders - flattened into plain descendants. Because clones can come from other documents (nested iframe capture), node-kind checks throughout use realm-safe predicates ([utils/is-element-node.ts](../src/utils/is-element-node.ts), [utils/is-html-element-of-tag.ts](../src/utils/is-html-element-of-tag.ts)) rather than `instanceof`, which fails across realms.

**Skipped and frozen elements.** `SKIPPED_CLONE_TAGS` (`script`, `style`, `link`, `head`, `slot`, …) are dropped. SVG template containers (`defs`, `symbol`, `pattern`, `mask`, gradients, filters) are cloned verbatim - they don't render directly, and their contents must stay untouched for references to resolve. Replaced elements are frozen individually:

- `<img>` is shallow-cloned with `src` pinned to `image.currentSrc`, which resolves the active `srcset`/`<picture>` candidate; the `srcset`, `sizes`, and `loading` attributes are in the removed-attribute list so the frozen candidate can't be re-negotiated (or lazily deferred) inside the capture. A lazy image that hasn't started loading has an empty `currentSrc`, so the `src` property is used, and a srcset-only pending image gets a candidate selected by [utils/select-srcset-candidate.ts](../src/utils/select-srcset-candidate.ts); the resource inliner then fetches the URL itself, so the capture shows the image even when the live element never decoded.
- `<canvas>` becomes an `<img>` with `canvas.toDataURL()` as its source, falling back to a transparent pixel when the canvas is tainted.
- `<video>` gets its current frame drawn to a scratch canvas and swapped in as an `<img>`, falling back to the `poster`. Because the UA gives `<video>` `object-fit: contain` while `<img>` defaults to `fill`, the element's computed `object-fit`/`object-position` are pinned inline on the substitute - the baseline diff was computed for a video, not an img.
- `<iframe>` becomes a `<div>` containing the nested capture's PNG (with the inner document's background color reproduced), or a flat gray placeholder for cross-origin/failed captures. Since a `<div>` defaults to `display: block` while an iframe is an inline replaced element, the emitted display is upgraded from `inline` to `inline-block` in [index.ts](../src/index.ts).

**Form state.** Live property state has no attribute reflection, so `reflectFormState` writes it back: `input.value` → `value` attribute, `input.checked` → `checked` attribute, `option.selected` → `selected` attribute, and `textarea.value` becomes the clone's text content. `checkbox.indeterminate` has no markup equivalent at all, so an indeterminate checkbox is styled into a pixel replica of Chromium's rendering ([utils/build-indeterminate-checkbox-style.ts](../src/utils/build-indeterminate-checkbox-style.ts)): `appearance: none` plus an inline SVG background of an accent-filled rounded rect with the centered white dash.

**XML sanitization.** The clone must survive `XMLSerializer` and re-parse as strict XML inside the SVG data URL. Attributes with non-XML names (Vue's `@click`, Alpine's `x-on:`, etc.) are removed, and [utils/strip-invalid-xml-characters.ts](../src/utils/strip-invalid-xml-characters.ts) scrubs control characters and lone surrogates from text and attribute values. Lone surrogates are _substituted_ with U+FFFD rather than deleted: browsers paint them as a visible replacement glyph, so removing them would change text layout while C0 controls (which are invisible) can simply be dropped.

## Notes about scroll wrappers and sticky descendants

Scroll offsets are not attributes; a serialized clone always renders at scroll position zero. For each element with a non-zero scroll offset and non-`visible` overflow, [capture/scroll-offsets.ts](../src/capture/scroll-offsets.ts) moves the clone's children into a wrapper `<div>` with `transform: translate(-scrollLeft, -scrollTop)`. The wrapper starts from `all: unset` and then sets `display` and every flex/grid formatting property to `inherit`, so a scroll container that is itself a flex or grid container keeps its inner formatting context - a plain block wrapper would re-lay-out flex children into a column.

That transform has a side effect that [capture/freeze-sticky.ts](../src/capture/freeze-sticky.ts) exploits: a transformed element becomes the containing block for absolutely positioned descendants. Sticky descendants of a scrolled container (which in the clone would render at their static position, since the clone doesn't scroll) are converted to `position: absolute` pinned at their _live painted offset_ expressed in the scroller's content coordinates - the wrapper's translation then nets out exactly, leaving the sticky element where the user saw it. A hidden placeholder with the sticky element's box and margins is inserted at its static position so siblings don't shift.

Fixed descendants have the mirror problem: a serialized clone has no browser viewport, so a kept `position: fixed` would resolve against the `foreignObject` box instead of where the element painted live. [capture/freeze-fixed.ts](../src/capture/freeze-fixed.ts) converts every fixed descendant to `position: absolute` at its live viewport-relative offset translated into the coordinates of the containing block it will get inside the clone - the nearest positioned/transformed ancestor, a scroll wrapper (using the same content-coordinate math as sticky), or the capture root itself. Fixed boxes are already out of flow, so no flow placeholder is needed.

## Notes about pseudo-element rules

Reading `getComputedStyle(element, "::before")` for every element triples the cost of the read pass on pages that don't use pseudo-elements at all. [capture/pseudo-elements.ts](../src/capture/pseudo-elements.ts) preflights the document's stylesheets once per capture (via [utils/visit-document-css-rules.ts](../src/utils/visit-document-css-rules.ts), which descends into `@import` and grouping rules and covers `adoptedStyleSheets`), looking for selectors that mention `:before`/`:after` or `:first-letter`. Only rule kinds that were found are snapshotted. The scan gives up after a 1000-rule budget or on any inaccessible (cross-origin) stylesheet and conservatively assumes all pseudo kinds exist. Elements inside shadow trees always get pseudo snapshots regardless - their styles come from shadow-root stylesheets the document-level scan never sees.

A `::before`/`::after` snapshot is only kept when its computed `content` is a real value (not `none`/`normal`). In the diff pass the pseudo styles are diffed against a sandbox baseline for the same tag + pseudo selector, and `content` is force-emitted regardless of the baseline - it's the property that makes the pseudo box exist, so it can never be dropped as "equal to default". Pseudo declaration blocks are part of the style-registry signature, so two elements only share a class when their pseudos match too. The `content` string is emitted verbatim, which is also why CSS counters work: `counter()`/`counters()` expressions and the elements' `counter-reset`/`counter-increment` properties (which survive the diff like any other painting property) re-evaluate natively over the cloned tree.

`::marker` follows the same preflight pattern: it is only read for elements whose `display` computes to a `list-item`, and only properties `::marker` can legally apply (font properties plus the small allowlist in `MARKER_STYLE_PROPS` from CSS Lists 3) survive. Like `::first-letter` it diffs against the element's own styles - markers inherit from their list item - except `content`, which is emitted verbatim whenever it isn't the default `normal`.

`::first-letter` is cheaper and stricter: only properties that can legally apply to a first-letter box (font, color, background, margins, borders, text decoration, …, per `FIRST_LETTER_STYLE_PROP_PREFIXES` in [constants.ts](../src/constants.ts)) are kept, and they're diffed against the element's _own_ styles rather than a sandbox baseline, since `::first-letter` inherits from the element itself.

## Notes about resource and font inlining

The serialized SVG loads as an isolated document with no network access from within an `<img>`, so every external reference must become a data URL first.

[capture/inline-resources.ts](../src/capture/inline-resources.ts) collects three kinds of references from the finished clone and registry: `<img src>`, SVG `<image href>` (both `href` and legacy `xlink:href`), and any `url(...)` inside registered declarations - which covers `background-image`, masks, `border-image-source`, pseudo `content`, and anything else, because the scan is value-driven rather than property-driven. `url()` tokens are parsed and rewritten by [utils/replace-css-urls.ts](../src/utils/replace-css-urls.ts), which handles quoting and escapes. All fetches go through [capture/resource-loader.ts](../src/capture/resource-loader.ts) and run under `Promise.allSettled`; a failed or timed-out fetch degrades to a transparent 1×1 pixel instead of failing the capture.

Fonts ([capture/embed-fonts.ts](../src/capture/embed-fonts.ts)) are filtered aggressively before any fetch: the used families are parsed out of the registered rules' `font-family` values (generic families excluded), and only same-document `@font-face` rules whose declared family is actually used - and which reference a `url()` at all - are rewritten and emitted. Relative font URLs resolve against the owning stylesheet's `href` (threaded through `visitDocumentCssRules` as the base URL), not the page URL. Cross-origin stylesheets (e.g. Google Fonts) hide their `cssRules` from the CSSOM entirely, so their hrefs are collected from the visitor's inaccessible-sheet callback, re-fetched as text with CORS, and their `@font-face` blocks recovered by a tolerant text parser ([utils/extract-font-face-blocks.ts](../src/utils/extract-font-face-blocks.ts)) before joining the same used-family filter and URL-inlining path.

SVG `<use>` is the odd one out because its targets often live entirely outside the captured subtree - a hidden sprite sheet at the end of `<body>`, or an external `.svg#icon` file. [capture/inline-svg-defs.ts](../src/capture/inline-svg-defs.ts) collects every fragment reference in the clone (`use href="#id"` plus `url(#id)` in attributes and in registered SVG paint properties), resolves each against the source document, clones the referenced element into a hidden `<svg><defs>` appended to the capture, and rescans each import for further references until the worklist drains - gradients referencing gradients, symbols referencing filters. External sprite files are fetched as text, parsed with `DOMParser`, and their fragments imported the same way, with the `use` href rewritten to the now-local `#fragment`.

## Notes about rasterization

[capture/rasterize.ts](../src/capture/rasterize.ts) is short but nearly every line is a workaround:

- **Data URL, not blob URL.** The natural choice for a large SVG string is `URL.createObjectURL(new Blob(...))`, but Chromium and WebKit taint canvases drawn from blob-URL SVGs that contain a `<foreignObject>` ([whatwg/html#10641](https://github.com/whatwg/html/issues/10641)) - `drawImage` succeeds and then `toBlob`/`toDataURL` throw. The equivalent `data:image/svg+xml` URL stays origin-clean, so the shared decoded image always loads from a data URL.
- **Decode once.** `decodeSvgImage` memoizes the decode promise so every output method shares one decoded `<img>`; a failed decode clears the memo so a retry is possible. `decoding = "sync"` is set to bias the decoder toward completing before paint.
- **Double rAF settle.** `img.decode()` resolving does not guarantee that nested data-URL resources _inside_ the foreignObject (inlined images, fonts) have finished compositing; drawing immediately produces captures with missing images. Two `requestAnimationFrame` ticks after decode let them settle.
- **Hidden-page fallback.** `requestAnimationFrame` never fires while the document is hidden (background tab, minimized window), which would hang the settle step forever. [utils/wait-for-animation-frames.ts](../src/utils/wait-for-animation-frames.ts) races each frame against a 100ms timeout so captures still complete from hidden pages.
- **16384px clamp.** Canvas dimensions are capped at 16384px per side (the common engine limit). Rather than clipping, a uniform `clampRatio` scales both the canvas size and the draw transform down, so oversized captures come out smaller but complete.

The optional `backgroundColor` is filled before the draw, `toBlob` encodes PNG via `canvas.toBlob`, and `toPngDataUrl` goes blob-first through a `FileReader` rather than `canvas.toDataURL`.

## Notes about caching and lifetimes

There are two lifetime tiers:

**Module-level, shared across captures.** The resource data-URL cache in [capture/resource-loader.ts](../src/capture/resource-loader.ts) is a FIFO map capped at 150 entries ([utils/create-fifo-cache.ts](../src/utils/create-fifo-cache.ts)), keyed by URL, with a companion in-flight map so concurrent captures of the same page dedupe fetches instead of racing them. The baseline cache in [capture/default-styles.ts](../src/capture/default-styles.ts) is a FIFO map capped at 30 entries keyed by tag/type/pseudo/font-size - baselines are UA behavior, not page state, so they're safe to share across captures and documents. The skip-verdict memo in [utils/should-skip-style-prop.ts](../src/utils/should-skip-style-prop.ts) is an unbounded map, bounded in practice by the universe of CSS property names. `activeCaptureDocuments` in [index.ts](../src/index.ts) is a module-level set that exists only to break iframe-capture recursion.

**Per-capture.** Everything else - the snapshot map, the style registry, the class-name and clone maps, and the sandbox iframe - is created inside `captureNode` and dies with it. The sandbox iframe is created lazily on the first baseline miss and removed in a `finally`, so a capture that never misses the baseline cache never touches the host DOM at all.

FIFO (not LRU) eviction is deliberate: both caches exist to absorb the burst of repeated lookups within a capture session, and FIFO gets that without per-hit bookkeeping.

## Notes about the fidelity harness

The harness ([e2e/fidelity.spec.ts](../e2e/fidelity.spec.ts)) treats the browser as its own oracle. For each fixture in [e2e/fixture-manifest.ts](../e2e/fixture-manifest.ts), a Playwright test loads the fixture page, stabilizes it ([e2e/utils/stabilize-page.ts](../e2e/utils/stabilize-page.ts): await `document.fonts.ready`, decode every image, inject `animation:none; transition:none; caret-color:transparent`, blur the active element), injects the IIFE bundle, captures `#target` with `captureNode({ pixelRatio: 1, scale: 1 })`, and takes Chromium's native `locator.screenshot()` of the same element in the same page state. Determinism comes from the Playwright config: fixed 1280×720 viewport, `deviceScaleFactor: 1`, `reducedMotion`, `--hide-scrollbars`, `--disable-lcd-text`, `--font-render-hinting=none`, single worker, zero retries.

Three assertions run per fixture, and each exists because the others have a blind spot:

- **Dimension delta.** The capture's PNG dimensions must be within 2px of the native screenshot's. This runs _before_ the images are cropped to their intersection - without it, a capture that renders at the wrong size would have its overflow silently cropped away and could still score perfectly.
- **Diff ratio.** Both PNGs are cropped to their intersection and compared with `pixelmatch` at color threshold 0.1 (anti-alias detection left on); the score is `diffPixels / (width × height)` and must stay under the fixture's `maxDiffRatio` budget. Budgets act as a ratchet: they are set to just above the observed score, so a regression on an already-perfect fixture fails immediately rather than drifting up toward a generous global tolerance. The strict budget is 0.005; a small class of fixtures (form controls, broken-image glyphs) gets 0.05 because native control chrome is platform-painted and legitimately diverges - each such fixture carries a manifest comment explaining why.
- **Mean channel delta.** The mean absolute per-channel (RGB) difference across the whole intersection ([e2e/utils/mean-channel-delta.ts](../e2e/utils/mean-channel-delta.ts)) must stay under 1 (8 for the form-control class). This closes pixelmatch's blind spot for uniform drift: a capture that is globally tinted, slightly transparent, or missing a subtle background stays under the 0.1 per-pixel color threshold on _every_ pixel and scores 0 on the diff ratio, while its mean delta lights up.

These blind spots were found by mutation-testing the harness during development - deliberately breaking capture stages (dropping a background, mis-sizing the root, skipping a style property) and checking that at least one fixture fails. Mutations that survived produced either a new assertion (the mean channel delta, the pre-crop dimension check) or a new fixture. The fixture corpus itself encodes the same idea at a larger scale: beyond the difficulty-coded core set (`01-solid-box` through `70-stress`), four families of regression fixtures are ported from the documented failure catalogs of prior-art libraries - `dtim-*` (dom-to-image-more), `hti-*` (html-to-image), `ms-*` (modern-screenshot), and `snap-*` (snapdom) - so every bug class those libraries hit is a permanent, budgeted test here, and a `lim-*` family exercises the limitation workarounds (marker styling, indeterminate checkboxes, offscreen lazy images and content-visibility subtrees, cross-origin font faces, fixed positioning under scroll). Fixtures whose live state depends on viewport proximity set `scrollTargetIntoViewBeforeScreenshot`, which scrolls `#target` into view and waits for image loads plus two frames after the capture, so the native ground-truth screenshot shows the fully loaded state the capture must reproduce. Every test attaches expected/actual/diff PNGs, and an `afterAll` hook writes all scores to `test-results/fidelity-scores.json` and prints a score table.

## Notes about the benchmark

[bench/benchmark.spec.ts](../bench/benchmark.spec.ts) runs `captureNode` head-to-head against snapdom, modern-screenshot, html-to-image, html2canvas, and dom-to-image-more ([bench/bench-manifest.ts](../bench/bench-manifest.ts)) on six fixtures from the fidelity corpus, using the same stabilized pages and the same deterministic browser config.

The methodology is built around two things benchmark suites usually get wrong:

- **Cold and warm are measured separately, on honest terms.** Each library's cold time is the median of 5 first-captures, each taken on a _fresh page_ (new `page`, fresh script injection), so no library benefits from another run's warmed caches. Warm time is the median and p95 of 8 back-to-back captures in one page, which is where this library's module-level caches (resources, baselines) legitimately count. Every capture runs under a 30s timeout plus a per-page budget so one hung library can't stall the suite.
- **Speed is reported next to fidelity.** Every benchmarked capture is also scored against Chromium's native screenshot with the same pixelmatch/mean-delta scoring the fidelity harness uses ([bench/utils/score-png-pair.ts](../bench/utils/score-png-pair.ts)), after a 2px dimension gate. A fast capture that renders the wrong pixels is not comparable to an accurate one, so the tables print fidelity score, mean delta, and PNG byte size alongside the timings.

The test only asserts that this library's own runs succeed - competitors are measured, not gated - and the full results are written to `test-results/bench-results.json`.
